\subsection{Rationale for Metric Selection in Static Analysis}
\label{sec:rationale_metrics}

The selection of evaluation metrics for static analysis tools requires careful consideration of the unique characteristics of security vulnerability detection. Static analysis for security must balance: comprehensive vulnerability coverage, minimization of false positives to maintain developer trust, and practical utility in real-world development environments.

\subsubsection{Why These Metrics for Static Analysis}
\label{sec:why_these_metrics}

The framework uses four main metrics, each chosen for its relevance to static analysis tool evaluation.

\textbf{Precision.} Precision is critical because false positives directly impact developer productivity and tool adoption. Research by \cite{Christakis2016} and \cite{Johnson2013} demonstrates that high false positive rates are among the primary barriers to static analysis tool adoption. In operational contexts, developers must manually investigate each flagged vulnerability; excessive false positives lead to alert fatigue. Precision answers: ``Can developers trust the tool's warnings?''

\textbf{Recall.} Recall is paramount in security contexts because false negatives represent undetected vulnerabilities that may be exploited. Missing a vulnerability can have severe consequences including data breaches, unauthorized access, and system compromise. Recall measures comprehensive security coverage, addressing: ``Does the tool find all vulnerabilities?''

\textbf{F1 Score.} The F1 Score provides a balanced assessment that accounts for both precision and recall. Static analysis tools must achieve high recall and acceptable precision; the F1 Score penalizes tools that excel in one dimension while failing in the other. It enables fair comparison between tools or configurations.

\textbf{Accuracy.} Accuracy provides an overall measure of correctness. Its interpretation in static analysis requires caution due to typically imbalanced datasets (vulnerabilities represent a small proportion of code). Accuracy remains useful when considered alongside precision and recall, particularly when comparing tools under similar evaluation conditions.

\subsubsection{Why F1 Score and Recall are Key Metrics}
\label{sec:f1_recall_key}

\textbf{Recall as a Primary Security Metric:} Recall is arguably the most important metric for security-focused static analysis because: undetected vulnerabilities (false negatives) pose direct security risks; security teams require comprehensive coverage; many standards and compliance frameworks require comprehensive vulnerability assessment; and high recall minimizes the probability that exploitable vulnerabilities remain undetected.

\textbf{F1 Score as a Balanced Performance Indicator:} The F1 Score serves as a key metric because: it ensures tools cannot optimize for one dimension at the expense of the other; tools with high F1 typically provide the best balance between security effectiveness and operational practicality; it enables fair comparison; the harmonic mean is more sensitive to low values; and it is a widely recognized standard in information retrieval and machine learning.

\subsubsection{Trade-offs in Static Analysis Metrics}
\label{sec:metric_tradeoffs}

\begin{itemize}
	\item \textbf{High recall, lower precision:} More false positives, but comprehensive coverage; often preferred in security-critical contexts.
	\item \textbf{High precision, lower recall:} Fewer false positives but may miss vulnerabilities; may be preferred when developer time is very limited.
	\item \textbf{Balanced approach (high F1):} Optimal balance for most operational contexts.
\end{itemize}

The selection of F1 Score and Recall as key metrics reflects the security-first nature of static analysis tools.

\subsection{Evaluation Metrics Overview}
\label{sec:metrics_overview}

The evaluation framework uses four main metrics from the confusion matrix, which groups predictions into four outcomes based on predicted vs.\ actual vulnerability states.

\subsubsection{Confusion Matrix Fundamentals}
\label{sec:confusion_matrix}

The confusion matrix categorizes predictions into four outcomes:

\begin{center}
\begin{tabular}{|l|c|c|}
	\hline
	& \textbf{Predicted: Vulnerable} & \textbf{Predicted: Safe} \\
	\hline
	\textbf{Actual: Vulnerable} & True Positive (TP) & False Negative (FN) \\
	\hline
	\textbf{Actual: Safe} & False Positive (FP) & True Negative (TN) \\
	\hline
\end{tabular}
\end{center}

\textbf{True Positive (TP):} Tool correctly identifies an actual vulnerability. \textbf{False Positive (FP):} Tool incorrectly classifies safe code as vulnerable; elevates false positive rates and consumes resources. \textbf{False Negative (FN):} Tool fails to detect an actual vulnerability; poses significant security risks. \textbf{True Negative (TN):} Tool correctly identifies safe code as non-vulnerable.

\subsection{Precision}
\label{sec:metric_precision}

Precision (Positive Predictive Value) quantifies the proportion of positive predictions that are correct: when the tool flags a location as vulnerable, what percentage are actual vulnerabilities?

\textbf{Formula:} Precision = TP / (TP + FP). Range 0--1: 1.0 = perfect (no false positives); 0.5 = half of positives incorrect; 0.0 = no value.

\textbf{Significance:} High precision is essential for developer trust and efficient resource utilization. \cite{Johnson2013} show that high false positive rates are a primary barrier to static analysis adoption.

\textbf{Illustrative calculation:} If the tool flags 100 locations and 74 are true vulnerabilities (TP=74, FP=26): Precision = 74/(74+26) = 0.74 (74\%).

\subsection{Recall (Sensitivity)}
\label{sec:metric_recall}

Recall (Sensitivity, True Positive Rate) quantifies the tool's ability to identify all actual vulnerabilities: of all genuine vulnerabilities, what proportion are detected?

\textbf{Formula:} Recall = TP / (TP + FN). Range 0--1: 1.0 = all detected; 0.5 = half missed; 0.0 = ineffective.

\textbf{Significance:} High recall is paramount in security; false negatives may be exploited. Practitioners often prioritize recall over precision, accepting more false positives for comprehensive coverage.

\textbf{Illustrative calculation:} If 100 actual vulnerabilities exist and the tool detects 93 (TP=93, FN=7): Recall = 93/(93+7) = 0.93 (93\%).

\subsection{F1 Score}
\label{sec:metric_f1}

The F1 Score is the harmonic mean of Precision and Recall, balancing both aspects and penalizing extreme imbalances.

\textbf{Formula:} F1 = 2 $\times$ (Precision $\times$ Recall) / (Precision + Recall). Range 0--1.

\textbf{Significance:} The harmonic mean is more sensitive to low values than the arithmetic mean. A tool with 100\% precision but 10\% recall yields F1 $\approx$ 0.18, reflecting poor overall performance.

\textbf{Illustrative calculation:} With Precision = 0.74 and Recall = 0.93: F1 = 2$\times$(0.74$\times$0.93)/(0.74+0.93) = 1.3764/1.67 $\approx$ 0.82.

\subsection{Accuracy}
\label{sec:metric_accuracy}

Accuracy measures the proportion of all predictions (positive and negative) that are correct.

\textbf{Formula:} Accuracy = (TP + TN) / (TP + TN + FP + FN). Range 0--1.

\textbf{Illustrative calculation:} With TP=74, TN=800, FP=26, FN=7: Accuracy = (74+800)/(74+800+26+7) = 874/907 $\approx$ 0.80 (80\%).

\subsection{Evaluation Results Analysis}

\subsubsection{SQL Injection Detection Performance}
\label{sec:eval_sqli}

Precision: 0.74 (74\%); Recall: 0.93 (93\%); F1: 0.82 (82\%); Accuracy: 0.80 (80\%).

The SQL Injection module demonstrates robust performance. Recall of 0.93 indicates the tool identifies 93\% of actual SQL injection vulnerabilities, which is significant for database security. Precision of 0.74 means 74\% of positive predictions are correct; the false positive rate is acceptable given the high recall. The F1 of 0.82 reflects a strong balance. The high recall is paramount for SQL injection, as undetected vulnerabilities can lead to severe breaches. Moderate precision suggests room for improvement via enhanced AST analysis and context-aware filtering.

\subsubsection{Cross-Site Scripting (XSS) Detection Performance}
\label{sec:eval_xss}

Precision: 0.50 (50\%); Recall: 0.40 (40\%); F1: 0.44 (44\%); Accuracy: 0.50 (50\%).

The XSS detection module exhibits moderate performance. The tool identifies less than half of actual XSS vulnerabilities and generates false positives for about 50\% of positive predictions. The low recall (0.40) is a significant security concern (60\% of XSS vulnerabilities undetected). The low precision can lead to alert fatigue. XSS complexity contributes to these challenges: multiple contexts (HTML, JavaScript, CSS, attributes), framework encoding, DOM-based XSS, and JavaScript context injection. Improved context-aware analysis and framework-specific sanitization recognition could improve performance.

\subsubsection{Command Injection Detection Performance}
\label{sec:eval_cmd}

Precision: 1.00 (100\%); Recall: 0.75 (75\%); F1: 0.85 (85\%); Accuracy: 0.87 (87\%).

The Command Injection module achieves perfect precision (1.00): every flagged finding is a true vulnerability with zero false positives. Recall of 0.75 indicates three-quarters of command injection vulnerabilities are detected. The F1 of 0.85 is the highest across all vulnerability types. Perfect precision suggests command injection patterns are highly distinctive for static analysis. The 25\% false negatives may stem from obfuscated patterns or less common vectors; improving recall while maintaining precision can make this module a model for others.

\subsubsection{Cross-Site Request Forgery (CSRF) Detection Performance}
\label{sec:eval_csrf}

Precision: 0.40 (40\%); Recall: 0.67 (67\%); F1: 0.50 (50\%); Accuracy: 0.33 (33\%).

The CSRF detection module exhibits the weakest performance. Recall of 0.67 indicates moderate success in identifying actual CSRF vulnerabilities, but precision of 0.40 means 60\% of positive predictions are false positives, significantly impacting practical utility. The accuracy of 0.33 indicates the tool incorrectly classifies two-thirds of code locations, suggesting fundamental challenges in distinguishing vulnerable from secure CSRF-related patterns.
