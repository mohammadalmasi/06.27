\chapter{Methodology}
\label{ch:methods}

This chapter presents the design and implementation of the proposed web-based vulnerability scanning platform. The methodology is divided into four main components: static analysis implementation, machine learning--based detection, web framework architecture, and the system workflow.

\section{Static-Based Detection}
\label{sec:static_detection}

This section presents how the static analysis engines in the platform detect vulnerabilities directly from Python source code without executing it. The core idea is to model how untrusted user input flows through the program until it reaches dangerous operations. For most injection vulnerabilities, the system uses taint analysis to follow data flows from sources (user input) to sinks (dangerous functions). For CSRF, which is caused by configuration and architectural flaws rather than data flows, the system uses a structural AST visitor that analyzes decorators, HTTP methods, and state-changing operations.

\subsection{Injection Vulnerabilities (Taint Analysis)}
\label{sec:static_taint}

For injection vulnerabilities such as SQL Injection, Cross-Site Scripting (XSS), and Command Injection, the static scanner performs taint analysis: it tracks how untrusted data moves through variables and expressions until it reaches a dangerous operation. If tainted data reaches a sink without proper sanitization, the scanner reports a vulnerability.

\subsubsection{Static Analysis Architecture}
\label{sec:static_arch}

The generic \texttt{TaintAnalyzer} implements a reusable architecture that can be configured
for different injection vulnerability types. It follows a fixed sequence of five steps:

\begin{enumerate}
  \item \textbf{Step 1: Parse the source code and build symbol tables.}
  The scanner first parses the Python source code into an Abstract Syntax Tree (AST).
  The AST encodes the syntactic structure of the program: assignments, function definitions,
  function calls, control flow, and so on. In addition to the AST, the scanner uses Python's
  built-in \texttt{symtable} library to build symbol tables. Symbol tables provide scope-aware
  information about variables and functions, ensuring that a variable named \texttt{query}
  in one function is treated separately from a \texttt{query} in another function. This scoping
  is crucial for avoiding false positives where taint in one function leaks into unrelated functions.

  \item \textbf{Step 2: Define taint sources and sinks.}
  The analyzer is configured with two lists, which differ by vulnerability type:
  taint sources (expressions that introduce user-controlled data, such as \texttt{request.form},
  \texttt{request.args}, \texttt{request.args.get(...)} or \texttt{input(...)}), and sinks
  (operations that can be dangerous when fed untrusted data). Example sinks include
  \texttt{cursor.execute(...)}, \texttt{cursor.executemany(...)}, and \texttt{cursor.raw(...)}
  for SQL injection; \texttt{render\_template\_string(...)} and \texttt{Markup(...)} for XSS;
  and \texttt{os.system(...)} or \texttt{subprocess.Popen(...)} with \texttt{shell=true} for
  command injection.

  \item \textbf{Step 3: Collect assignments and sink calls with scope context.}
  Using the AST and symbol tables, the analyzer walks the syntax tree and records each
  assignment (variable, right-hand-side expression, and scope identifier) and each sink
  call (function, arguments, and scope identifier). By keeping this information scope-aware,
  the analyzer can reason about data flows function by function and avoid incorrectly
  connecting unrelated variables.

  \item \textbf{Step 4: Propagate taint through assignments and expressions.}
  The analyzer determines which variables hold user-controlled data within each scope.
  If the right-hand side of an assignment is a taint source, then the left-hand side variable
  becomes tainted in that scope. When an expression combines tainted and untainted values
  (for example, string concatenation with \texttt{+} or f-strings), the result is also tainted.
  Known sanitization functions such as \texttt{int()} or \texttt{escape()} remove taint from
  their outputs. This propagation step is repeated over all recorded assignments until a
  fixpoint is reached, meaning that running another pass does not introduce any new tainted
  variables.

  \item \textbf{Step 5: Check each sink and report vulnerabilities.}
  For each sink call, the analyzer checks whether any of its arguments are marked as tainted
  in the current scope and have not been sanitized. If so, the scanner reports a vulnerability.
  Each finding includes the line number, the type of vulnerability, a short description, and
  a code snippet with remediation advice.
\end{enumerate}

\subsubsection{Example: SQL Injection}
\label{sec:static_sql_example}

The following code shows a typical SQL injection vulnerability:

\begin{lstlisting}[style=pycode]
def get_user():
    user_id = request.form["user_id"]  # user input (source)
    query = "SELECT * FROM users WHERE id = '" + user_id + "'"  # tainted
    cursor.execute(query)  # sink: executes tainted query
\end{lstlisting}

The analysis proceeds as follows:
\begin{enumerate}
  \item The AST and symbol tables isolate the variables \texttt{user\_id} and \texttt{query} inside the \texttt{get\_user} function.
  \item \texttt{request.form} is treated as a taint source, and \texttt{cursor.execute} is treated as a sink.
  \item At the assignment of \texttt{user\_id}, the variable is marked tainted. At the assignment of \texttt{query}, the concatenation with \texttt{user\_id} propagates the taint to \texttt{query}.
  \item When the analyzer encounters \texttt{cursor.execute(query)}, it sees that \texttt{query} is tainted and reports a SQL injection vulnerability at that line.
\end{enumerate}
Because user input flows directly into the argument of \texttt{cursor.execute()}, this represents a textbook source-to-sink injection vulnerability that the generic taint analyzer is designed to detect.

\subsection{Cross-Site Request Forgery (CSRF) Analysis}
\label{sec:static_csrf}

CSRF vulnerabilities are fundamentally different from injection vulnerabilities: there is no single untrusted value flowing into a dangerous function. Instead, CSRF issues arise from configuration and architectural flaws, such as disabling built-in protections or executing state-changing operations on unsafe HTTP methods. For this reason, taint analysis alone is insufficient for CSRF, and the scanner uses a dedicated AST visitor that focuses on structure rather than variable data flows.

\subsubsection{Why generic taint analysis is insufficient for CSRF}
\label{sec:static_csrf_limitations}

Two representative scenarios illustrate why a pure taint-based approach cannot reliably detect CSRF flaws:

\begin{enumerate}
  \item \textbf{Disabling CSRF protections via decorators.}\\
  Many frameworks (e.g., Django) enable CSRF protection by default. A common vulnerability
  is to explicitly disable it with a decorator:

\begin{lstlisting}[style=pycode]
from django.views.decorators.csrf import csrf_exempt

@csrf_exempt  # Vulnerability: CSRF protection disabled
def update_password(request):
    new_pass = request.POST.get("password")
    request.user.set_password(new_pass)
    request.user.save()
\end{lstlisting}

  Here, the vulnerability is caused by the presence of the \texttt{@csrf\_exempt} decorator
  itself, not by a dangerous data flow. A generic taint analyzer, which only tracks variables,
  does not naturally capture this pattern.

  \item \textbf{State changes on GET requests.}\\
  Another class of CSRF bugs appears when state-changing operations are allowed on
  \texttt{GET} routes:

\begin{lstlisting}[style=pycode]
@app.route('/transfer', methods=['GET'])  # Only GET allowed
def transfer_money():
    amount = request.args.get('amount')
    user = get_current_user()
    user.balance -= int(amount)
    user.save()  # State change on GET (vulnerability)
\end{lstlisting}

  In this case, the data flow from \texttt{amount} to \texttt{user.balance} is not inherently
  unsafe. What makes it vulnerable is the context: a state-changing database update executed
  in a route restricted to \texttt{GET}. Taint analysis does not understand HTTP verb semantics
  and thus cannot detect this on its own.
\end{enumerate}

\subsubsection{Custom CSRF AST visitor}
\label{sec:static_csrf_visitor}

To handle CSRF, the scanner uses a custom AST visitor that focuses on structural properties of route handlers:

\begin{itemize}
  \item \textbf{Decorator analysis:} for each function definition, the visitor examines its decorators. If a decorator name such as \texttt{csrf\_exempt} or \texttt{disable\_csrf} appears, the scanner flags the function as vulnerable and records a finding.
  \item \textbf{Context-aware state change analysis:} the visitor inspects route definitions to determine whether a handler is restricted to \texttt{GET} methods only. If so, it walks the function body and looks for method calls that indicate state changes, such as \texttt{save()}, \texttt{delete()}, \texttt{update()}, or \texttt{commit()}. Any such combination (state-changing call inside a \texttt{GET}-only route) is reported as a CSRF vulnerability.
\end{itemize}

By combining decorator inspection with context-aware scanning of state-changing operations, the CSRF visitor can detect configuration and architectural flaws that are outside the scope of standard taint analysis.

\subsection{Scanner Execution Interface}
\label{sec:static_execution_api}

Both the taint analyzers and the CSRF AST visitor are exposed through a common execution API. The backend can:
\begin{itemize}
  \item Scan raw source code strings (e.g., when the user pastes code in the UI).
  \item Scan files on disk.
  \item Fetch code from URLs (e.g., GitHub raw links) and scan the downloaded content.
  \item Return structured JSON results that include summary statistics, line-level findings, and highlighting information for the frontend and report generator.
\end{itemize}

\subsection{Limitations and Future Work}
\label{sec:static_limitations}

The current static scanners are highly effective for intra-procedural analysis (within a single function or class) and for structural checks such as CSRF decorators and HTTP methods. However, building a full compiler-grade static analysis engine in pure Python introduces several limitations:
\begin{itemize}
  \item \textbf{Inter-procedural analysis:} if tainted data is passed from one function into a helper function (for example, \texttt{execute\_query(tainted\_data)}), the current analyzer loses track of taint after the function boundary. Robust inter-procedural analysis requires constructing a global call graph across files.
  \item \textbf{Dynamic and library-driven behavior:} when tainted data interacts with dynamic constructs (e.g., \texttt{eval()}, \texttt{getattr()}) or complex third-party libraries, a static AST-based approach struggles to predict runtime behavior, leading to potential false negatives.
\end{itemize}

To address these limitations, a promising future direction is to integrate established static analysis engines such as Semgrep, Bandit, or Pysa/Pyre. In such an architecture, the custom definitions of sources, sinks, and structural CSRF checks developed in this work can be expressed as rules for an external engine. The Python backend invokes the external tool, parses its structured JSON output, and feeds it into the existing frontend, machine learning pipeline, and reporting modules, achieving whole-codebase analysis while preserving the domain-specific vulnerability logic of the thesis.

\section{Machine Learning--Based Detection}
\label{sec:ml_detection}

This section presents the step-by-step process of how the machine learning scanner
detects SQL injection vulnerabilities, followed by concrete examples to show how these
steps apply in practice. This structure gives a clear overview of the theory first, making
the examples easy to follow.

\subsection{How the Scanner Works}
\label{sec:ml_part1}

\begin{enumerate}
  \item \textbf{Step 1: Tokenization.}\\
  The scanner reads the source code file and breaks it down into a flat list of tokens
  (words, symbols, operators, and so on).

  \item \textbf{Step 2: Sliding Windows.}\\
  The model does not process the whole file at once. Instead, it uses a \emph{sliding window}
  approach to scan through the token list:
  \begin{itemize}
    \item \texttt{WINDOW\_LENGTH = 200}: the model looks at a block of 200 tokens at a time.
    \item \texttt{WINDOW\_STEP = 5}: the window moves forward by 5 tokens to create the next block.
  \end{itemize}
  Each window heavily overlaps with the previous one. The model evaluates each window
  independently.

  \item \textbf{Step 3: Vectorization (Tokens $\rightarrow$ Vectors $\rightarrow$ Matrix).}\\
  For each window, the tokens must be converted into numerical representations that the
  neural network can understand:
  \begin{enumerate}
    \item Each token is looked up in a pre-trained Word2Vec model to get a vector of 300
    numbers (for example, \texttt{"query"} $\rightarrow$ \([0.12, -0.45, 0.87, \ldots]\)).
    Unknown tokens become vectors of 300 zeros.
    \item The vectors are stacked into a matrix of shape \texttt{[window\_length, 300]}.
    \item If the window has fewer than 200 tokens (for example at the end of a file or for a
    very short file), it is padded with zeros up to exactly 200 rows.
    \item The matrix is wrapped in a batch dimension, resulting in an input tensor
    \texttt{X} of shape \texttt{(1, 200, 300)} (one sample, 200 time steps, 300 features per step).
  \end{enumerate}

  \item \textbf{Step 4: Model Prediction.}\\
  The prepared matrix \texttt{X} is fed into the ML model (a BiLSTM neural network). The
  model outputs a single probability score \texttt{prob} between 0 and 1 representing the
  likelihood that this specific 200-token window contains an SQL injection pattern.

  \item \textbf{Step 5: Threshold Check and Line Reporting.}\\
  The scanner checks the prediction against a confidence threshold (default 0.5):
  \begin{itemize}
    \item If \texttt{prob >= 0.5} (for example \texttt{0.97}), it flags a vulnerability.
    \item The line number reported is the line where the first token of that window is located.
  \end{itemize}
  \textit{Note:} the model does not pinpoint the exact line of the bug. It simply states:
  ``The 200-token window that starts at this line looks suspicious.''

  \item \textbf{Step 6: Deduplication.}\\
  Because the window slides by only 5 tokens at a time, a single SQL injection block
  is often captured by multiple overlapping windows. If multiple windows trigger a
  vulnerability and happen to start on the exact same line, the scanner deduplicates them
  and keeps only the report with the highest confidence score for that line.
\end{enumerate}

\subsection{Examples in Practice}
\label{sec:ml_part2}

Now we trace some actual code through these steps.

\begin{enumerate}
  \item \textbf{Example 1: A Short File (Single Window).}\\
  \textbf{Source Code.}

\begin{lstlisting}[style=pycode]
def vulnerable_sql_high_1():
    """Direct string concatenation in SELECT"""
    user_id = request.form["user_id"]
    query = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(query)
\end{lstlisting}

  \textbf{Walkthrough.}

  \textbf{1.~Tokenize.} The scanner turns the whole file into a list of tokens (words and
  symbols). For this code it is roughly:

\begin{table}[H]
  \centering
  \begin{tabular}{lll}
    \textbf{Token index} & \textbf{Token} & \textbf{Line} \\
    \hline
    0  & \texttt{def}                   & 1 \\
    1  & \texttt{vulnerable\_sql\_high\_1} & 1 \\
    2  & \texttt{(}                     & 1 \\
    3  & \texttt{)}                     & 1 \\
    4  & \texttt{:}                     & 1 \\
    5  & \texttt{user\_id}              & 3 \\
    6  & \texttt{=}                     & 3 \\
    7  & \texttt{request}               & 3 \\
    \multicolumn{3}{c}{\dots} \\
    33 & \texttt{query}                 & 5 \\
    34 & \texttt{)}                     & 5 \\
  \end{tabular}
  \caption{Illustrative token indices and line numbers for Example~\ref{sec:ml_example1}.}
\end{table}

  So you get one long list of about 35 tokens.

  \textbf{2.~Sliding windows.} Since the total number of tokens (about 35) is less than
  \texttt{WINDOW\_LENGTH} (200), there is only one window (tokens 0 to 35).

  \textbf{3.~Vectorize.} The 35 tokens are converted to 300-dimension vectors and padded
  with zeros up to length 200. The input tensor \texttt{X} has shape \texttt{(1, 200, 300)}.

  \textbf{4.~Prediction.} The model evaluates this single window and outputs one number,
  for example \texttt{prob = 0.97}.

  \textbf{5.~Threshold \& line reporting.} Since \texttt{0.97 >= 0.5}, a vulnerability is
  reported. The window starts at token index 0, which maps to line~1.

  \textbf{6.~Deduplication.} Only one report exists, so deduplication does nothing.

  \textbf{Result.} A single vulnerability is reported for the file at line~1.

  \item \textbf{Example 2: A Longer File (Multiple Overlapping Windows).}\\
  \textbf{Source Code.}

\begin{lstlisting}[style=pycode]
def vulnerable_sql_high_1():
    """Direct string concatenation in SELECT"""
    user_id = request.form["user_id"]
    query = "SELECT * FROM users WHERE id = '" + user_id + "'"
    cursor.execute(query)

def vulnerable_sql_high_2():
    """F-string with SQL"""
    name = request.args.get("name")
    query = f"SELECT * FROM users WHERE name = '{name}'"
    cursor.execute(query)
\end{lstlisting}

  \textbf{Walkthrough.}

  \textbf{1.~Tokenize (visualizing the tokens and lines).} Imagine the tokens are numbered.
  Each token lives on a line:

\begin{verbatim}
token 0  -> line 1   -> "def"
token 1  -> line 1   -> "vulnerable_sql_high_1"
...
token 10 -> line 7   -> "def"
token 11 -> line 7   -> "vulnerable_sql_high_2"
token 12 -> line 7   -> "("
token 13 -> line 9   -> "name"       <- SQL code starts here
token 14 -> line 10  -> "query"
token 15 -> line 11  -> "cursor"
...
\end{verbatim}

  \textbf{2.~Sliding windows (\texttt{step = 5}).} The sliding window (step = 5) creates the
  following windows:

\begin{verbatim}
Window 1: starts at token 0  (line 1)  -> model gets tokens 0..199
Window 2: starts at token 5  (line 3)  -> model gets tokens 5..204
Window 3: starts at token 10 (line 7)  -> model gets tokens 10..209
Window 4: starts at token 13 (line 9)  -> model gets tokens 13..212  <- SQL code inside
Window 5: starts at token 14 (line 10) -> model gets tokens 14..213  <- SQL code inside
Window 6: starts at token 15 (line 11) -> model gets tokens 15..214  <- SQL code inside
\end{verbatim}

  \textbf{3 \& 4.~Vectorize \& prediction.} Windows 4, 5, and 6 all contain the SQL injection
  code (because the window is 200 tokens wide and they all overlap). The model gives
  them high probabilities such as:

\begin{verbatim}
Window 4 -> prob 0.97
Window 5 -> prob 0.97
Window 6 -> prob 0.969
\end{verbatim}

  \textbf{5.~Threshold \& line reporting.} Because all three are at least 0.5, the scanner
  reports three vulnerabilities. It reports the start line of each window:
  \begin{itemize}
    \item Window 4 reports a bug at line 9.
    \item Window 5 reports a bug at line 10.
    \item Window 6 reports a bug at line 11.
  \end{itemize}

  \textbf{6.~Deduplication.} Since these windows start on different lines (9, 10, and 11),
  the line-based deduplication does not merge them.

  \textbf{Result.} You get multiple reports on consecutive lines. All these windows contain
  the same SQL injection code because they overlap; the only difference is where they
  start. You get multiple reports for different lines not because the model found separate
  bugs, but because overlapping windows all saw the same bug and each one reported its
  own start line.
\end{enumerate}

\subsection{One-Sentence Summary}
\label{sec:ml_summary}

\begin{quote}
The scanner tokenizes the file, slides a 200-token window across it (moving 5 tokens at a
time), converts each window into a numerical matrix, asks the model for a probability,
and reports the start line of any window scoring \texttt{prob >= 0.5}.
\end{quote}

\section{Web Framework Architecture}
\label{sec:web_framework}

The platform is implemented using a Flask backend and a React frontend, providing a responsive and user-friendly interface~\cite{React,Flask,AlmasiToolRepo,AlmasiDeployedApp}. The system follows a three-tier architecture: \textbf{Presentation Layer (Frontend)} --- React-based web application for user interface and interaction; \textbf{Application Layer (Backend)} --- Flask-based REST API server handling business logic and analysis; \textbf{Analysis Layer} --- Static analysis scanners and machine learning models for vulnerability detection.

\subsection{Frontend (React)}
\label{sec:frontend}

The frontend is built with React and TypeScript, providing a component-based, responsive user interface. Key components include: \textbf{Scanner} --- for selecting vulnerability type (SQL Injection, XSS, Command Injection, CSRF), analysis mode (static or ML), and input method (paste, file upload, GitHub URL); \textbf{Results} --- for displaying vulnerabilities with code highlighting, severity indicators, filtering, and report download. The frontend validates input, prepares JSON payloads, sends HTTP POST requests to the backend API, and renders results with highlighted code and summary metrics. Communication uses the Fetch API; CORS is enabled for cross-origin requests between frontend and backend.

\subsection{Backend (Flask)}
\label{sec:backend}

The backend is a Python Flask REST API server that receives requests, routes them to the appropriate scanner or ML analyzer, and returns JSON responses~\cite{Flask,AlmasiToolRepo}. For static analysis, endpoints include \texttt{/api/scan-sql-injection}, \texttt{/api/scan-xss}, \texttt{/api/scan-command-injection}, \texttt{/api/scan-csrf}; for machine learning, \texttt{/api/scan-ml} handles all vulnerability types with a \texttt{type} parameter. Report generation endpoints (\texttt{/api/generate-*-report}) produce Word documents using python-docx. The backend handles code upload and parsing, static analysis scanning using AST-based techniques, ML model inference, result aggregation, and error handling (validation, timeouts, GitHub URL fetching).

\subsection{Components}
\label{sec:web_components}

\textbf{Code Submission.} Supports direct input (paste), file uploads, and GitHub repository scanning. Users can provide Python source code through any of these methods. The frontend validates that input is not empty, file size does not exceed limits, and GitHub URLs are properly formatted.

\textbf{Analysis Engine.} Routes code to either static or machine learning detection modules based on user selection. The backend exposes RESTful API endpoints for each vulnerability type and analysis mode. Request processing validates input, maps vulnerability types to scanner modules or ML model modes, and executes the appropriate analysis pipeline.

\textbf{Results Visualization.} Interactive frontend highlights vulnerable code, shows vulnerability type and severity, and allows download of Word reports with remediation suggestions and CWE/OWASP references. Results include vulnerability list, code highlighting (red for vulnerable, green for safe), severity breakdown, and filtering by severity level.

\subsection{Design Goals}
\label{sec:design_goals}

\begin{itemize}
	\item \textbf{Modular and scalable architecture} --- Separate scanner modules per vulnerability type; three-tier design enables independent updates and extensions.
	\item \textbf{Easy integration of additional detection methods} --- Architecture supports adding new scanners or ML models without major refactoring.
	\item \textbf{Open-source availability} --- Ensures accessibility for students, developers, and researchers who cannot afford commercial security tools.
\end{itemize}

\section{System Workflow}
\label{sec:system_workflow}

This section presents how to use the system. From the user's perspective, the workflow is the same for both static analysis and machine learning: the user selects an analysis mode, provides code, and receives results. The only differences occur internally during preprocessing and vulnerability detection. The system workflow consists of the following steps:

\subsection{Code Submission}
\label{sec:workflow_submission}

User provides Python code via one of the supported methods: direct paste, file upload, or GitHub repository URL. The frontend validates input (non-empty, file size limits, valid URL format) and sends the request to the backend. This step is identical for both static and ML modes.

\subsection{Preprocessing}
\label{sec:workflow_preprocessing}

The backend preprocesses the code according to the selected mode:
\begin{itemize}
	\item \textbf{Static analysis:} Parse into AST using Python's \texttt{ast} module.
	\item \textbf{Machine learning:} Tokenize and encode for the BiLSTM model (character-level one-hot or Word2Vec embeddings).
\end{itemize}

\subsection{Vulnerability Detection}
\label{sec:workflow_detection}

The analysis engine runs the appropriate detector:
\begin{itemize}
	\item \textbf{Static analysis:} Apply rule-based checks (regex patterns, AST taint analysis, sink detection).
	\item \textbf{BiLSTM model:} Predict vulnerabilities and compute probability scores.
\end{itemize}

\subsection{Result Compilation}
\label{sec:workflow_compilation}

Detected vulnerabilities are organized into a report with line numbers, types, and mitigation suggestions. Results are deduplicated, severity-classified (Critical, High, Medium, Low), and formatted as JSON for the frontend. This step is the same for both modes.

\subsection{User Interface Display}
\label{sec:workflow_display}

Frontend presents results interactively, allowing users to review and compare outputs from both methods. Users can filter by severity, view highlighted code (red for vulnerable, green for safe), download Word reports, and switch between static and ML results for comparison.
